7. Cross-encoder rerank

The top 10 fused candidates are re-scored by a model that reads the query and each chunk together.

Up to here, the query and chunks were embedded separately and compared by distance. That is a bi-encoder: fast, because every chunk vector is precomputed, but it never sees the query and a chunk at the same time. The reranker is a cross-encoder: it takes each (query, chunk) pair as one input and outputs a relevance score. It is far more accurate and far more expensive.

The two-stage pattern. Running the cross-encoder on all chunks for every query would be too slow: it is a full model forward pass per pair, not a vector lookup. So cheap retrievers (vector + BM25, fused by RRF) narrow the field to 10 candidates, and the expensive model only ranks those. Note the reordering: a chunk the cross-encoder reads as directly answering the question can jump several places, and the defining fact_orders.md chunk lands in the final set.
